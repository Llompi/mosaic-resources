# Contributing Guidelines

Thank you for considering contributing to this community resource directory!

## ⚠️ Before You Contribute

Please understand:
- **No guarantees** — Contributions may or may not be accepted
- **No timelines** — Reviews happen when maintainers are available
- **Volunteer project** — This is maintained in spare time with no commitments

## Ways to Contribute

### Report Outdated Information
If you find incorrect, outdated, or broken resources:
1. [Open an Issue](../../issues/new)
2. Include:
   - City and category name
   - Resource name
   - What's wrong (closed, moved, incorrect link, etc.)
   - Correct information if known

### Add New Resources
To suggest a new resource:
1. [Open an Issue](../../issues/new) first to discuss
2. Or submit a [Pull Request](../../pulls) with:
   - Organization/resource name
   - Official website URL
   - Brief description (1-2 sentences)
   - Verification that it's currently active
   - Which city and category it belongs in

### Propose New Cities/States
1. [Open an Issue](../../issues/new)
2. Include:
   - City/state name
   - Why it should be added
   - At least 5-10 initial resources you can provide
   - Willingness to help research resources

### Fix Bugs or Improve Code
1. Fork the repository
2. Create a feature branch (`git checkout -b feature/improvement-name`)
3. Make your changes
4. Test thoroughly
5. Submit a Pull Request with clear description

## Resource Quality Standards

New resources should:
- ✅ Be from official organizations (government, nonprofits, established services)
- ✅ Have working websites with current information
- ✅ Serve vulnerable populations or provide essential services
- ✅ Be free or low-cost
- ❌ Not be commercial services (unless no free alternatives exist)
- ❌ Not be personal blogs or unverified sources

## Code Standards

- Keep all code in a single `index.html` file
- Maintain existing CSS variable naming (`--rw-*` prefix)
- Follow existing JavaScript patterns
- Ensure accessibility (ARIA labels, keyboard navigation)
- Test on mobile and desktop

## Pull Request Process

1. Ensure code works locally
2. Update documentation if needed
3. Add details to your PR description:
   - What changed
   - Why it's needed
   - How you tested it
4. Be patient—reviews happen when available
5. Be open to feedback and changes

## Questions?

Open an [Issue](../../issues/new) with the `question` label.
